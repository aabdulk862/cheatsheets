# HTTP Questions

# **What is HTTP?**

- **HTTP (HyperText Transfer Protocol)** is the backbone of internet communication, enabling data exchange between clients (e.g., browsers) and servers.
- It's essential for full-stack applications, as it supports interaction between the front-end and back-end through **HTTP requests** and **HTTP responses**.

## **HTTP Request**

- **Method/Verb**: Specifies the action (e.g., `GET`, `POST`, `PUT`).
    - **Example:** `GET /users/1 HTTP/1.1` (Fetch user with ID 1).
- **URL**: The endpoint or resource location. Example: `/users`, `/orders`.
- **Header**: Metadata about the request (e.g., authorization, content type).
    - **Example:**
    
    ```bash
    Authorization: Bearer <token>
    Content-Type: application/json
    ```
    
- **Body**: Data sent to the server (used with methods like POST/PUT).Example:
    
    ```json
    {
        "name": "John Doe",
        "email": "john.doe@example.com"
    }
    ```
    

## **HTTP Response**

- **Status Code**: Indicates the result of the request.Examples:
    - `200 OK`: Request succeeded.
    - `404 Not Found`: Resource not found.
    - `500 Internal Server Error`: Server error.
- **Header**: Metadata about the response (e.g., content type).
    - **Example:**
    
    ```bash
    Content-Type: application/json
    ```
    
- **Body**: Contains the response data or error message.Example:
    
    ```json
    {
        "id": 1,
        "name": "John Doe",
        "email": "john.doe@example.com"
    }
    ```

# **What is an API?**

- An **API (Application Programming Interface)** is a set of rules and protocols that allow different software applications to communicate with each other.
    - Enables interaction between different software systems, hiding the implementation details while exposing only necessary functionalities
- **Types of API’s**
    - **Web APIs**: Use HTTP (e.g., REST, SOAP) for communication between a client (like a browser or app) and a server.
    - **Library APIs**: Allow integration of third-party libraries or modules in your code.
- **Uses**:
    - Accessing data or services (e.g., fetching weather data or making payment transactions).
    - Integrating systems (e.g., using Google Maps API in an app).

# **What is a "REST API"?**

- **REST** (REpresentational State Transfer) is an architectural style for building web services. It provides a set of guidelines to structure server-side resources and enables communication between clients and servers over the web.
- **REST is an Architectural Style**:
    - It is not a language, framework, or design pattern.
    - REST outlines guidelines for organizing and interacting with server-side resources.
- **Resources**:
    - A resource represents data or an entity (e.g., a user, a product).
    - Each resource is uniquely identified by a URI (Uniform Resource Identifier).
    - **Example:** `/users` represents all users.
- **REST** relies on specific **HTTP methods** to perform actions on resources:
    - **GET, POST, PUT, PATCH, DELETE**

# **What are HTTP verbs?**

- **REST** relies on **HTTP methods (verbs)** to interact with server resources. Each method serves a specific purpose, aligning with **CRUD operations** (Create, Read, Update, Delete) in databases.
- **GET**:
    - **Purpose**: Retrieve information from the server.
    - **Characteristics**: Safe, idempotent, empty body.
    - **Example**: `GET /users` retrieves all users.
- **POST**:
    - **Purpose**: Create a new resource or submit data.
    - **Characteristics**: Data in request body, not idempotent.
    - **Example**: `POST /users` adds a new user.
- **PUT**:
    - **Purpose**: Update an entire resource.
    - **Characteristics**: Replaces all data, idempotent.
    - **Example**: `PUT /users/1` replaces all data for user with ID 1.
- **PATCH**:
    - **Purpose**: Update specific fields of a resource.
    - **Characteristics**: Partial updates, idempotent.
    - **Example**: `PATCH /users/1` updates specific attributes (e.g., email).
- **DELETE**:
    - **Purpose**: Remove a resource.
    - **Characteristics**: Idempotent in most implementations.
    - **Example**: `DELETE /users/1` deletes the user with ID 1.

# **What are HTTP status codes?**

- **HTTP status codes** are part of the server's response to indicate the outcome of a client's request. They are grouped into five categories, each representing a different type of response.

### **1xx: Informational**

- Rarely used, provides request status updates.
    - **101 Switching Protocols**: Server switches protocols (e.g., HTTP to WebSocket).

### **2xx: Success**

- Indicates successful processing of the request.
    - **200 OK**: Request succeeded.
    - **201 Created**: Resource created successfully.
    - **202 Accepted**: Request accepted, processing asynchronously.
    - **204 No Content**: Request succeeded, no content returned.

### **3xx: Redirects**

- Additional action required to complete the request.
    - **301 Moved Permanently**: Resource has a new permanent URL.
    - **302 Found**: Temporary redirection.
    - **307 Temporary Redirect**: Temporary redirection, same method used.

### **4xx: Client Errors**

- Indicates issues in the client’s request.
    - **400 Bad Request**: Malformed or invalid request.
    - **401 Unauthorized**: Missing or invalid authentication.
    - **403 Forbidden**: Authenticated but lacks permission.
    - **404 Not Found**: Requested resource not found.
    - **418 I’m a Teapot**: Fun status refusing coffee in a teapot.
    - **420 Enhance Your Calm**: Rate-limiting response (non-standard).

### **5xx: Server Errors**

- Server issues encountered while processing the request.
    - **500 Internal Server Error**: Generic server issue.
    - **501 Not Implemented**: Functionality not supported.
    - **502 Bad Gateway**: Invalid response from a gateway or proxy.
    - **504 Gateway Timeout**: Gateway or proxy timed out.

# **What is Postman used for?**

- Postman is a popular tool for testing and debugging HTTP requests to servers. It provides a user-friendly interface to interact with APIs and is particularly useful in developing and testing backend services.
- **Making HTTP Requests**:
    - Postman allows you to send various types of HTTP requests (GET, POST, PUT, DELETE, etc.) to test endpoints.
    - You can include headers, query parameters, body content (like JSON or form data), and authorization details.
- **Testing Servers**:
    - Postman is ideal for testing Java servers or APIs before building a frontend.
    - It ensures that the server responds with the correct data and HTTP status codes.
- **Troubleshooting Full-Stack Applications**:
    - If a request fails from the frontend but succeeds in Postman, the issue is likely in the frontend.
    - If the request also fails in Postman, the issue is likely in the backend.
