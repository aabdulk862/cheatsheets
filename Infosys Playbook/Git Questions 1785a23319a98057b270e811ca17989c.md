# Basic commands (pushing to repo, pulling from repo)

1. **git clone**: Copies a remote repository to your local machine, allowing you to work on the project locally.
    
    ```bash
    git clone <repository-url>
    ```
    
2. **git add .**: Stages all changes in the current directory (and its subdirectories) to be committed.
    
    ```bash
    git add .
    ```
    
3. **git commit -m "commit message"**: Commits the staged changes to your local repository, with a descriptive message about the changes.
    
    ```bash
    git commit -m "Added feature X"
    ```
    
4. **git push**: Pushes your local commits to a remote repository (e.g., GitHub), making your changes available to others.
    
    ```bash
    git push
    ```
    
5. **git pull**: Fetches the latest changes from a remote repository and merges them into your local repository.
    
    ```bash
    git pull
    ```
    
6. **git status:** Shows the current state of your working directory and staging area.
    
    ```bash
    git status
    ```
    
7. **git branch:** Lists branches in your repository, creates new ones, or deletes them.
    
    ```bash
    git branch                  # List branches  
    git branch <name>           # Create a branch  
    git branch -d <branch-name> # Deletes branch
    ```
    
8. **git checkout:** Switches to a different branch or commit.
    
    ```bash
    git checkout <branch-name>    # Switch to a branch  
    git checkout -b <new-branch>  # Create and switch to a branch
    ```
    
9. **git merge:** Merges changes from one branch into another.
    
    ```bash
    git merge <branch-name>
    ```
    
10. **git log:** Shows the commit history for the repository.
    
    ```bash
    git log
    ```
    

# What is a branch?

- **Git Branching** enables developers to work on features, bug fixes, or experiments in isolation, keeping the main production code stable. Branching facilitates parallel development, version management, and collaboration.
- **Key Concepts:**
    - **Main Branch**: The default, stable branch (e.g., `main` or `master`) representing the working version of the project.
    - **Branching**: Allows developers to create separate timelines for new features or fixes without impacting the main codebase.
    - **Merging**: Combines changes from a branch back into the main branch or another branch to integrate new features or fixes.
    - **Snapshot Representation**: A branch serves as a timeline of changes specific to a feature or fix.
- **Benefits:**
    - **Isolation**: Ensures independent development without disturbing the main branch.
    - **Collaboration**: Enables multiple team members to work concurrently on different branches.
    - **Experimentation**: Allows testing of ideas or features without risking the stable codebase.
    - **Efficient Merging**: Simplifies integrating completed work into the main branch.

# How to merge a branch?

- Merging a branch combines the changes from one branch into another, typically integrating a feature or fix into the `main` branch. Follow these steps:
    1. **Switch to the Target Branch**
        - Ensure you’re on the branch where you want to merge changes (e.g., `main`).
            
            ```bash
            git checkout main
            
            ```
            
    2. **Merge the Source Branch**
        - Use the `git merge` command to combine the source branch into the target branch.
            
            ```bash
            git merge <source-branch>
            ```
            
    3. **Resolve Merge Conflicts (if any)**
        - If there are conflicting changes, Git will pause the merge and highlight the conflicts. Edit the conflicting files to resolve the issues, then stage the resolved files.
            
            ```bash
            git add <file>
            ```
            
    4. **Complete the Merge**
        - Once all conflicts are resolved, complete the merge by committing the changes (if Git didn’t do this automatically).
            
            ```bash
            git commit
            ```
            
- **Example Workflow:** If merging a feature branch (`feature-login`) into `main`:
    
    ```bash
    git checkout main         # Switch to the main branch
    git pull                  # Get the latest changes
    git merge feature-login   # Merge the feature branch into main
    ```

# What is CI/CD?

- **CI/CD/CD** represents the processes of **Continuous Integration (CI)**, **Continuous Delivery (CDelivery)**, and **Continuous Deployment (CDeployment)** in the DevOps lifecycle.
- These practices aim to streamline software development, testing, and deployment by automating workflows. CI/CD/CD helps teams deliver reliable and frequent updates while minimizing errors and reducing manual intervention.

### **Continuous Integration (CI)**

- **Purpose**:
    
    CI ensures that developers regularly merge their code changes into a central repository, which is then automatically built and tested. This process detects and resolves issues early, preventing integration conflicts.
    
- **Key Processes**:
    1. Developers write code and push it to a shared repository (e.g., GitHub or GitLab).
    2. Automated tools build the application to ensure code compatibility.
    3. Unit tests and integration tests validate that the new code works with existing functionality.
- **Benefits**:
    - Early detection of bugs reduces costs and delays.
    - Encourages collaboration by integrating code frequently.
    - Maintains a constantly updated version of the application.
- **Automation Tools**:
    - **Git**: Version control for managing code repositories.
    - **CI Tools**: Jenkins, GitLab CI/CD, CircleCI.
    - **Testing Frameworks**: JUnit, Selenium.
- **Metaphor**:
    
    A pizza chef (developer) assembles a pizza (application) and sends it to the quality control team (automated CI system) to check for issues before proceeding.
    

---

### **Continuous Delivery (CDelivery)**

- **Purpose**:
    
    Builds on CI by automating the steps required to prepare code for production. The application is automatically built, tested, and deployed to a staging or testing environment. Human intervention is needed for deployment to production.
    
- **Key Processes**:
    1. After CI, successful builds are promoted to staging or testing environments.
    2. Staging environments are used for further testing, such as:
        - **End-to-End Testing**: Verifying the entire workflow.
        - **Performance Testing**: Ensuring the application meets speed and resource requirements.
        - **User Acceptance Testing**: Validating features with real or simulated users.
    3. Once tests pass, the artifact is ready for manual deployment to production.
- **Benefits**:
    - Reduces deployment risk by testing in environments similar to production.
    - Speeds up the release process while maintaining control.
    - Ensures that every build is production-ready.
- **Automation Tools**:
    - **Pipelines**: Jenkins, GitLab CI/CD, AWS CodePipeline.
    - **Testing Tools**: JMeter, Postman, Cypress.
- **Metaphor**:
    
    A delivery driver (pipeline) brings the pizza (application) to your door (staging). You check the pizza for quality before accepting it and serving it to your guests (users).
    

---

### **Continuous Deployment (CDeployment)**

- **Purpose**:
    
    Extends Continuous Delivery by automating the deployment to production. If the staging tests pass, code changes are automatically pushed to production without human intervention.
    
- **Key Processes**:
    1. After CDelivery, the application undergoes final automated tests in staging.
    2. If tests pass, the code is automatically deployed to production.
    3. Changes are made available to end users almost immediately after a developer pushes code.
- **Benefits**:
    - Enables rapid iteration and frequent releases.
    - Reduces human error during deployment.
    - Supports high-velocity development environments, like those used by major platforms (e.g., Gmail or Facebook).
- **Automation Tools**:
    - Jenkins, GitLab CI/CD, Spinnaker, AWS CodeDeploy.
- **Metaphor**:
    
    A pizza cannon (pipeline) shoots the pizza (application) directly to your table (production) as soon as it passes all quality checks. The chef (developer) and user (customer) don’t need to do anything after the cannon fires.
